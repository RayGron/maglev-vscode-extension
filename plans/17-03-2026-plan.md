# Maglev VS Code Extension Plan

## Goal

Develop `maglev-vscode-extension` into a panel-first VS Code extension with a full local-agent workflow.

## Product Plan

1. Build a dedicated chat/webview panel instead of relying on modal prompts.
2. Move the task flow into the panel: task entry, plan review, edit review, checks, commit review, push, and deploy.
3. Add a message bridge between the webview and the extension host for all user actions.
4. Improve edit review with per-file status, diff opening, and selective application directly from the panel.
5. Improve commit review with editable `title/body` fields in the panel.
6. Improve deploy review with host, branch, remote command, and result logs in the panel.
7. Add live run logs and step status updates while a run is active.
8. Add persistent run history inside the extension.
9. Unify panel UX with inline completion, code actions, and selection-driven tasks.
10. Finish release polish for manifest, branding, and packaging assets.

## Backlog

### Phase 1: Webview Shell

1. Add a dedicated webview provider
   Create a module such as `apps/vscode-extension/src/ui/chatView.ts` or `apps/vscode-extension/src/ui/chatPanel.ts`.
2. Update [extension.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/extension.ts)
   Register the panel command and connect it to the existing runtime.
3. Keep current task flow working
   Use the existing runtime as the execution backend while the UI migrates to the panel.

### Phase 2: Extension Host to Webview Bridge

1. Add a message protocol module
   Create a file such as `apps/vscode-extension/src/ui/messages.ts` for request and response payloads.
2. Update [extension.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/extension.ts)
   Wire webview messages to runtime actions.
3. Reuse existing services
   Continue using [backendStatus.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/services/backendStatus.ts), [editReview.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/services/editReview.ts), and [deployReview.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/services/deployReview.ts) as the host-side helpers.

### Phase 3: Panel-Based Task Flow

1. Replace modal task entry in [taskPanel.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/ui/taskPanel.ts)
   The panel should collect the task and show live run state.
2. Expand [runView.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/ui/runView.ts)
   Adapt it from a static markdown summary into data for a persistent UI.
3. Add run-state rendering
   Show plan steps, proposed edits, applied edits, commit message, commit hash, and deploy result.

### Phase 4: Edit Review in Panel

1. Keep diff preview support from [editReview.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/services/editReview.ts)
   Expose it through webview actions instead of modal selection only.
2. Add file list state to the panel
   Show each proposed edit with summary and current status.
3. Preserve selective application
   Keep the existing subset-apply behavior already supported by the runtime.

### Phase 5: Commit Review in Panel

1. Keep generation in the runtime
   Reuse the existing commit preparation flow already wired through the runtime.
2. Replace input boxes in [taskPanel.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/ui/taskPanel.ts)
   Move commit title/body editing into the panel UI.
3. Reflect final commit state in [runView.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/ui/runView.ts)
   Show the reviewed message and resulting commit hash.

### Phase 6: Deploy Review in Panel

1. Keep deploy preview generation
   Reuse [deployReview.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/services/deployReview.ts) as the host-side approval helper until the panel fully owns that step.
2. Add deploy block in the panel
   Show `host`, `sshCommand`, `remoteCommand`, and result logs.
3. Update [extension.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/extension.ts)
   Expose a panel-driven `deployLatest` path in addition to the command palette entry.

### Phase 7: Product Integration

1. Connect panel state to [codeActionProvider.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/providers/codeActionProvider.ts)
   Let code actions open or seed the panel instead of only firing direct commands.
2. Keep inline completion in [inlineCompletionProvider.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/providers/inlineCompletionProvider.ts)
   Use the same backend configuration and status information as the panel.
3. Keep runtime config centralized in [settings.ts](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/src/config/settings.ts)
   Continue to load model/backend defaults from [model-endpoints.json](/mnt/e/dev/Repos/maglev-vscode-extension/config/model-endpoints.json).

### Phase 8: Release Polish

1. Add panel assets and build integration
   Ensure webview assets are bundled by the extension build scripts in `scripts/`.
2. Improve manifest polish in [apps/vscode-extension/package.json](/mnt/e/dev/Repos/maglev-vscode-extension/apps/vscode-extension/package.json)
   Finalize branding, categories, descriptions, and commands.
3. Keep packaging green
   Continue validating `npm run build:debug`, `npm run build:release`, and `.vsix` packaging.

## Order of Execution

1. Webview shell
2. Host-to-webview bridge
3. Panel task flow
4. Panel edit review
5. Panel commit review
6. Panel deploy review
7. Product integration
8. Release polish
